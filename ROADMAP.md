# IAM Access Atlas — Production-Readiness Roadmap

> Companion to `CLAUDE.md`. Phase-by-phase plan to take the site from "working" to "production-ready and impressive." All four phases are in scope.

**Guiding principles**
- Zero-build, zero-framework, zero-CDN. Vendor any dependency.
- Preserve existing personas (Manager / Staff) and visual identity.
- Every change must keep the site openable as a local file (`file://`) — no required dev server.
- Verify after each phase: cross-browser smoke (Chrome/Safari/Firefox), mobile breakpoints, no console errors.

## Phase 2 — UX & Accessibility (target: WCAG 2.1 AA)

**Goal:** Make the site usable by everyone, on every device, with every input method.

### 2.1 Keyboard & focus
- [ ] Skip-to-content link as the first focusable element (`<a href="#app-content" class="skip-link">Skip to content</a>`)
- [ ] Focus trap inside `#stat-modal` while open (cycle Tab/Shift-Tab inside)
- [ ] Restore focus to the trigger element on modal close
- [ ] Global `Escape` already closes modal — extend to also collapse the staff compare panel
- [ ] Keyboard shortcut: `/` focuses the role search input (when not already in an input)
- [ ] Visible `:focus-visible` outline on **all** interactive elements (currently only some have it)

### 2.2 ARIA & semantics
- [ ] Add `aria-live="polite"` region for the result count (announces as user types)
- [ ] Add `<caption class="sr-only">` to the roles table describing what it lists
- [ ] Add `aria-sort` to sortable headers (if sorting is added)
- [ ] Filter pills: use `role="tablist"` / `role="tab"` + `aria-selected`
- [ ] Announcements banner: `role="status"` or `role="alert"` depending on type
- [ ] All icon-only buttons must have `aria-label` (audit existing — most do, verify all)
- [ ] Add `lang` attributes where content shifts language (none today, but watch)

### 2.3 Color contrast (WCAG 2.1 AA = 4.5:1 normal text, 3:1 large)
- [ ] Audit lavender `#7C8EC8` on white backgrounds — likely fails for body text, OK for large/UI
- [ ] Audit `.modal-app-meta` `#8A8A8A` on white — borderline; bump to `#6B6B6B` or darker
- [ ] Audit `.empty-sub` `#8A8A8A` — same concern
- [ ] Audit hero tagline `rgba(255,255,255,0.45)` on `#12121A` — likely fails; lift to 0.6
- [ ] Audit footer `#A0A0A0` on `#EDEBF5` — likely fails
- [ ] Use a tool (axe DevTools, Lighthouse, or `npx pa11y http://localhost:8000`) for objective measurement

### 2.4 Motion preferences
- [ ] Wrap all animations in `@media (prefers-reduced-motion: no-preference) { … }`, OR
- [ ] Add `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; } }` reset
- [ ] Skip the animated stat counter — set value immediately when this preference is set
- [ ] Skip the hero opacity scroll fade

### 2.5 Print stylesheet polish
- [ ] Add a print-only header line showing: App name, generated date, generated by
- [ ] Ensure table doesn't break rows across pages (`tr { page-break-inside: avoid; }`)
- [ ] Hide row hover backgrounds in print
- [ ] Force admin/covered/risk badges to render with high-contrast monochrome (printers may suppress backgrounds)

### 2.6 Graceful error/loading states
- [ ] If `data.js` fails to load, show a clear message with a "report this" mailto link instead of silent failure
- [ ] If a search returns 0 results, the existing "No roles match" state is good — verify it announces to screen readers (`aria-live`)
- [ ] If `APP_DATA.applications` is empty, show a sensible empty state instead of an empty dropdown

**Acceptance criteria for Phase 2**
- axe-core scan: 0 critical, 0 serious violations
- Keyboard-only navigation works for every interactive element including modals
- All text passes WCAG 2.1 AA contrast
- Site is usable and looks intentional at 320px viewport width
- `prefers-reduced-motion: reduce` disables all non-essential animation

## Phase 3 — Insights Dashboard (the wow factor)

**Goal:** Add a third view that lets the IAM team and stakeholders see catalog-wide patterns at a glance.

### 3.1 Vendor Chart.js
- [ ] Download Chart.js v4 UMD build → save to `vendor/chart.umd.min.js` (single file, ~60 KB)
- [ ] Include with `<script src="vendor/chart.umd.min.js"></script>` before `app.js`
- [ ] Note in `vendor/README.md`: source URL, version, license (MIT), date downloaded

### 3.2 Add Insights as a third mode
- [ ] On the landing page, add a third mode card alongside Manager/Staff: **Insights** (icon: chart bars)
- [ ] Description: "Catalog-wide view of role distribution, PHI coverage, and access-risk patterns. For IAM, compliance, and leadership."
- [ ] New view section `#insights-content` with grid of chart cards

### 3.3 Charts to build (priority order)

1. **KPI tile strip** (top of insights view)
   - Total applications | Total roles | PHI-covered apps | Admin roles | High-risk roles
   - Each tile shows count + percentage where meaningful

2. **PHI coverage donut**
   - Apps with at least one PHI-covered role vs. apps with none
   - Color: amber for covered, sage for not covered, grey for unknown

3. **Risk distribution stacked bar (or horizontal bar)**
   - Counts of Standard / Sensitive / Elevated / High Risk roles across the catalog
   - Use the existing badge colors for visual continuity

4. **Top 10 apps by role count** (horizontal bar)
   - Helps identify catalog complexity hotspots

5. **Admin density** (horizontal bar)
   - Top 10 apps by admin-to-total-role ratio (only apps with ≥3 roles to avoid noise)
   - Governance signal: which apps have disproportionate admin footprint

6. **Catalog hygiene flags** (table card, not a chart)
   - Apps with 0 roles in `Roles.csv`
   - Apps where every role has `covered: null` (PHI classification missing)
   - Apps where every role has `admin: null`
   - Each flag clickable → navigates to that app's detail view

### 3.4 Implementation notes
- Charts should respect `prefers-reduced-motion` (Chart.js supports `animation: false`)
- All charts must have `<canvas>` `aria-label` describing the data
- Below each chart, render a hidden but accessible data table (`<table class="sr-only">`) for screen readers
- Tooltips should show absolute count and percentage
- Color palette: reuse existing badge colors for continuity

**Acceptance criteria for Phase 3**
- Insights view opens from landing page
- All 6 visualisations render with real data from `APP_DATA`
- Charts re-render on browser resize
- Screen reader can read chart data via the sr-only table
- No Chart.js CDN reference — file is vendored

## Phase 4 — Optional polish

**Goal:** Quality-of-life features that elevate it from "good internal tool" to "delightful internal tool."

### 4.1 Deep-link routing
- [ ] URL parameter `?mode=manager|staff|insights&app=Okta&role=Administrator`
- [ ] On page load: parse params, jump to that view/app/role
- [ ] On mode/app change: `history.pushState` to update URL without reload
- [ ] On browser back/forward: `popstate` listener restores state
- [ ] Enables shareable links inside Slack DMs, UAR campaign emails, Jira tickets

### 4.2 Per-app PDF export (UAR evidence pack)
- [ ] Add a "Download PDF" button next to Print/Copy in Manager actions
- [ ] Use `window.print()` with `@page` CSS + a dedicated print layout that includes:
  - App name, role count, generation date
  - Reviewer name field (blank line)
  - Approval signature field
  - Full role table
  - Footer with site URL and "Generated on YYYY-MM-DD"
- [ ] No JS PDF library needed — print-to-PDF is universal

### 4.3 Dark mode (optional)
- [ ] Add a small theme toggle in the app nav bar
- [ ] CSS custom properties for color palette in `:root` + `[data-theme="dark"]`
- [ ] Persist preference in `localStorage` (key: `iam_atlas_theme`)
- [ ] Default to `prefers-color-scheme`

**Acceptance criteria for Phase 4**
- Pasting a deep-link URL into a fresh tab opens directly to that app/role
- Browser back/forward navigates view state correctly
- "Download PDF" produces a clean reviewer-ready document
- Dark mode (if shipped) has parity with light mode for contrast

## Out of scope (explicitly deferred)

- Authentication / Okta SSO integration (decision: site stays public)
- Automated CSV refresh from ServiceNow (decision: stay manual)
- Jira API integration for live UAR data (decision: use stub file for now)
- Test framework / CI pipeline (overhead not justified at current scale)
- Framework migration (React, Vue, etc.) (zero-build philosophy holds)
- Backend / API layer (data stays embedded)

If any of these become priorities later, they're additive — none require rework of Phases 1–4.
